The three ducks of redux

  • First: single source of truth.
  • state application single object tree. Not split. access state, access whole tree
  • why? serialized easily. universal apps, easy to hydrate your state on client.
  • A single state makes it easier to debug or inspect your state
  • 2nd: State is read-only
  • Change emit an action, plain javascript object describing the change.
  • 3rd: Changes made with pure function. What pure functions are? The results is determined purely on the input parameters, without any side effects.
  • We write 'state reducers' as pure functions.
  • Together we will refer to these as the three ducks of redux

    Actions

    Plain JS object describing a change to the state

    { type: 'ADD_TODO', text: 'Draw a duck' }
{ type: 'TOGGLE_TODO', index: 1 }
{ type: 'SET_VISIBILITY_FILTER', filter: 'SHOW_ALL' }

    ...but not describing how to change the state

  • So we emit actions, as plain javascript objects, describing the change we want to make to our state
  • Describe WHAT is changing, but not how to change it
  • The only mandatory field on and action is a type field, which needs to be string
  • Our actions completely serializable, normal javascript objects. This is important for later

    Reducers

    A pure function that take the previous state and an action and returns a new state 

    function myReducer(state = [], action) {
    switch(action.type) {
        case "ADD_TODO":
            return { /* ... */ }
        default:
            return state;
    }
}
  • A reducer if a PURE function that takes in previous state and an action, and returns a new state.
  • Every action will go through every reducer
  • Common to put a switch
  • duck 2, our state is completely read only.if we want to make changes to the state, we make a copy, modify the copy, and return it as our new state.
  • default case, and just return our state

    The store

    import { createStore, combineReducers } from 'redux';

const reducers = combineReducers({
  reducer1, reducer2
});

const store = createStore(reducers);
store.dispatch(
    { type: 'ADD_TODO', text: 'Draw a duck' }
);

const unsubscribe = store.subscribe(listener);
const state = store.getState();
  • interact redux via store -> holds state
  • When we create we pass the store about all the reducers
  • holds the state, access it via the getState() function
  • dispatch actions against it via the dispatch function
  • register listeners against it via the subscribe function. These listeners will be called every time, and action is dispatched against our store

Reacting to state changes

  • We could add listeners to our state, and get values via store.getState() every time our state changes
  • More likely to use existing bindings like react-redux
  • Having a well managed state great, but how do we read data out of it
  • We could subscriber to state changes, and dig into our state each time it changes and update our UI
  • But that is a fairly low level API, and we can use existing bindings from libraries like react-redux to make the state available to our application
  • Which uses some black magic to make our store available to all child components

WebSockets

  • Two way communication between browser and server
  • Lower overhead than http
  • Uses same ports as http (80 and 443 for TLS)
  • Widely supported*


*https://caniuse.com/#feat=websockets

  • now we knwo reux, So on to part 2, what are websockets
  • 2 way TCP comm between the client (in this case the browser) and the server
  • send any serializable data between the client and the server. Like a json payload. Or a redux action. Or your entire redux state tree. See where Im going with this?
  • lower overhead
  • same ports at http, meaning we wont have to open up any additional ports in out firewall
  • Websockets are also pretty widely supported, roughly 94% of browsers in use today support them

socket.io - Server

const app = require('express')();
const server = require('http').Server(app);
const io = require('socket.io')(server);

server.listen(3000);

io.on('connection', (client) => {
    /* ... */
});
  • Socket.io is library that makes use of websockets, and makes it really easy for us to get up and running in just a few lines of code
  • To keep things easy we will use use node and express for our server side
  • So lets see how we configure Socket.io on our webserver
  • Create an express app, create an http server from our express app, and bind our socket.io instance to it.
  • now we can look for 'connection' events, and pass through a function that will be called with a reference to the client that just connected. So this will be called every time a client connects

Server

/* ... */
io.on('connection', (client) => {
    client.emit('private_message',
            'Welcome, you are now connected');

    io.emit('broadcast_message',
            'A new user has connected');

    client.on('disconnect', () => {
        io.emit('broadcast_message',
            'A user has disconnected');
    });
});
  • Socket.io has a couple of predefined events, such connection and disconnect
  • We can also define our own event types that we can define when we emit a message and when when we listen for messages
  • When a client is connected we can choose to either send a message directly to the client, liek here where we end a welcome message
  • Or we can broadcast a message to all connected clients, for example letting them know that a new user has joined.

Client

import io from 'socket.io-client';

const socket = io();

socket.on('connect', () => {
    console.log("Connected to server");
    socket.emit('private_message',
        'Hello server')
});

socket.on('disconnect',
    () => console.log("Disconnected"));
  • For the client side, we will need to import the client library for socket.io, socket.io-client
  • Instantiate it, and very similar to the server side, we can listen for the connect event (different to the connection event on the server)
  • and we can emit data using the emit function.
  • We can t do any peer to peer communication, so anything emitted from the client goes directly to the server
  • Socket.io can do quite a few other cool things, like broadcast to specific user, and create chat rooms for users to join, but we won't be using those features today
